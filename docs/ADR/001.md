# ADR-0001: osmcli（MVP）の技術選定とアーキテクチャ

- Status: Accepted（提案）
- Date: 2026-01-27
- Decision owner: osmcli team（yuiseki）

## 背景 / コンテキスト

osmcli は、Nominatim / Overpass API / Valhalla API を、AIエージェントが繰り返し合成できる「非対話・決定論的I/O」のCLIとして束ねる。
CLIはUNIXパイプラインを前提に、stdoutはデータのみ、stderrはログのみ。データフォーマットは json / geojson / jsonl（ndjson）を中心とする。

特にMVPでは以下を最重要とする。

- I/Oの安定性（機械可読、壊れにくい、スキーマ安定）
- 大きい結果でも破綻しにくい（ストリーミング、メモリ効率）
- 曖昧性・空結果・上流障害を「分類して」機械処理できる
- TDD前提で、テスト容易な構造（境界の明確化、依存の注入、HTTPモック）

---

## 決定ドライバ

1. 開発スピード（実装と変更の速さ、学習コストの低さ）
2. 信頼性（I/O規律、エラー分類、リトライ容易性）
3. パイプライン親和性（stdin/stdout、JSONL、1行処理）
4. テスト容易性（HTTPモック、CLI統合テスト、純粋関数化）
5. 将来の差し替え（OSMプロファイルが主役だが抽象化は維持）

---

# 決定

## 1) ランタイムとモジュール方針

- Node.js は v24 LTS を基準（最低でも v22 以上）。
  根拠: Node公式のリリース状況で v24 が Active LTS として更新されているため、MVPの基準に据えるのが最も安全。
- TypeScript は ESM（`"type": "module"`）前提。
- CIでは `node --version` を固定し、スキーマ・出力の決定論性を担保する。

## 2) CLIフレームワーク

- commander.js を採用する（サブコマンド・オプション・helpの実装速度が高い）。
  commanderは引数パース、usageエラー、helpシステムなどCLIの基本を提供する。
- コマンド体系はPRD通り `geocode`, `reverse`, `aoi resolve`, `poi count`, `poi fetch`, `route`, `isochrone`, `doctor` をトップレベルに配置。

## 3) データ形式とI/O規律（UNIX最適化）

- 大量データの標準形式は JSON Lines（拡張子 .jsonl、別名NDJSON）を第一級として扱う。
  JSON Lines は「1行=1JSON」で行単位処理ができ、UNIXパイプやログ、プロセス間連携に適する。
- GeoJSONはRFC 7946 を準拠仕様とする。
- stdin/stdoutはNodeストリームで扱い、行単位処理は `node:readline` を用いる。
  `node:readline` はReadable（例: stdin）から1行ずつ読むインターフェイスを提供する。
- stdoutはデータのみ、stderrはログのみ（PRD準拠）。例外として「`--format json` の場合は失敗時もstdoutへ機械可読JSONを出す」ことで、エージェントのパース成功率を優先する（終了コードは非0、ログはstderr）。

### 出力フォーマットの基本

- `--format json`: 単一JSON（集計・メタ・doctor・route等）
- `--format geojson`: Feature / FeatureCollection
- `--format jsonl`（または `ndjson` 表記を許可）: 1行1Feature（poi fetchのデフォルト推奨）

## 4) エラー分類と終了コード

- 終了コードを固定し、機械処理を容易にする（PRD要件を満たす最小集合）。

提案コード（例）:

- 0: success
- 2: invalid_input（引数不正・stdin不正）
- 3: not_found（空結果）
- 4: ambiguous（候補が複数で決められない）
- 5: upstream_temporary（タイムアウト・429・5xx等、リトライ可能）
- 6: upstream_permanent（4xx等、恒久失敗）
- 7: internal_error（バグ・想定外）

JSONエラー（stdoutへ出す場合）例:

```json
{
  "error": {
    "code": "AMBIGUOUS",
    "message": "multiple candidates for place query",
    "hints": { "use": ["--limit 1", "--country JP", "--lang ja", "--all"] },
    "candidates": [
      /* optional */
    ]
  }
}
```

## 5) 上流APIアクセス（HTTPクライアント）

- undici を採用する。
  Nodeの組み込み `fetch` はUndiciが基盤である一方、Undiciを直接使うと接続プール設定や MockAgent 等、組み込みfetchでは不足しがちなAPIが使える。
- 「ホストごとにPoolを保持」し、コマンド間の単発実行でも同一プロセス内の複数リクエストはコネクション再利用する（doctorや複合コマンドで効く）。
- タイムアウト・リトライはアダプタ層で統一実装し、全コマンドで同じ挙動にする。

## 6) 大きいJSONの取り回し（Overpass想定）

- Overpassなど巨大JSONを扱う可能性があるため、配列丸読み前提は置かない。
- MVPでは以下の方針で段階導入する。

段階A（最速で実装）

- 取得結果が小さいケースは通常 `JSON.parse` で扱う（geocode等）。

段階B（POI fetchなど大きくなりうる箇所）

- `stream-json` を採用し、HTTPレスポンスをストリームでパースして Feature を逐次生成し、jsonlとして逐次stdoutへ吐く。
  `stream-json` は「最小メモリフットプリントでJSON処理パイプラインを組む」ためのストリーム部品を提供する。

## 7) キャッシュ（冪等・再実行容易性）

- ローカル永続キャッシュとして cacache を採用する。
  cacache は高速で並行性に強い、content-addressableなディスクキャッシュを提供し、npm自体のHTTPキャッシュ基盤でもある。
- キャッシュキーは「プロバイダ種別 + エンドポイントURL + 正規化入力 + 重要オプション（lang/country/mode等）」のハッシュ。
- TTLを設定可能（例: 24h）。上流障害時の再試行を減らし、エージェントループでの速度と安定性を上げる。

## 8) 入出力バリデーション（型安全と壊れにくさ）

- 外部入力（stdin・上流APIレスポンス・設定ファイル）には Zod を用いてランタイム検証する。
  ZodはTypeScript-firstのスキーマ宣言とバリデーションを提供し、`parse` で入力検証できる。
- 「出力スキーマの安定」をADRで約束するため、レスポンスの最終整形直前にZodで整合性チェックをかける（異常時は internal_error 扱い）。

## 9) TDDとテスト戦略（徹底）

- テストランナーは Vitest を採用する（TypeScript開発での立ち上げが早く、並列実行などで高速化しやすい）。
- テストピラミッドを固定する。

1. ユニットテスト（最重要）
   - 正規化（地名、タグ、preset、--within解決など）
   - 例外/エラー分類（終了コード・JSONエラー）
   - フォーマッタ（json/geojson/jsonl）

2. アダプタテスト（HTTP境界）
   - UndiciのMockAgent/MockPoolを使い、Nominatim/Overpass/Valhallaのレスポンスを再現してテストする。
     MockPoolの例（intercept→reply 等）がドキュメントに示されている。
   - 加えて、MSW (Mock Service Worker) を採用し、より宣言的なHTTPモックを実現する。
     MSWはリクエストインターセプトをService Worker風のAPIで提供し、統合テストでのHTTPモックを容易にする。
3. CLI統合テスト（E2Eの最小）
   - `execa` でCLIをサブプロセス実行し、stdout/stderr/exit code を丸ごと検証する。
     execaは `child_process` をベースに「プログラム用途に最適化されたコマンド実行」を提供する。

## 10) 開発体験と配布

- 開発時の実行: tsx を採用（ビルド無しでTSを実行でき、開発ループが短い）。Node公式ドキュメントでもTS runnerとして紹介されている。
- 配布ビルド: tsup を採用（esbuildベースで設定少なく高速にバンドル）。
- Linter/Formatter: Biome を採用する（Rust製で高速、ESLint+Prettierの代替として単一ツールで完結）。
- MVP配布形態は npmパッケージ（`bin` で `osmcli` を提供）を第一にする。将来、単一実行ファイル化（SEA等）は別ADRで検討。

## 11) Geo Processing

- 地理空間演算（バッファ、交差判定、面積計算、座標変換など）が必要になる場合、Turf.js を採用する。
  Turf.jsはGeoJSON操作のための包括的なライブラリで、豊富な地理空間関数を提供する。
- MVPでは `--within` の形状判定やisochroneの結果整形などで利用する可能性がある。

# アーキテクチャ（MVP）

## レイヤ構造

- presentation（CLI）
  - commanderで引数解釈、help、終了コード制御

- application（usecase）
  - 1コマンド=1ユースケース
  - 入力正規化、キャッシュ、リトライ、エラー分類の中心

- domain（ports / types）
  - Geocoder / AOIResolver / POIFinder / Router などのインターフェイス定義

- infrastructure（providers）
  - OSMプロファイル実装（Nominatim/Overpass/Valhalla）
  - undici、stream-json、cacache をここに閉じ込める

- io（formatters / readers）
  - stdin（`-`）/ `@file` ロード
  - json/geojson/jsonl エンコード（逐次出力）

## データフロー例（poi fetch --format jsonl）

1. `--within` を解決（地名→AOI、またはGeoJSON入力）
2. Overpassへクエリ
3. レスポンスをストリーミング解析（stream-json）
4. Featureへ変換し、1行ずつstdoutへ書き出し（JSON Lines）

このときJSON Linesの要件（1行1JSON、UTF-8、改行区切り）を満たすことで、`jq` 等の後段が安定する。

# 代替案（却下理由）

- 対話的TUI（inquirer等）
  PRDの「非対話」が最重要のため却下。
- すべてをGeoJSON FeatureCollectionで返す
  大規模結果でメモリを圧迫しやすく、jsonl（逐次）に劣るため。JSON Linesは行単位処理を前提にできる。
- 組み込みfetchのみを使用
  Node fetchはUndici基盤だが、MockAgent等の周辺APIや細かな制御が不足しうるため、テスト容易性と制御性の観点でundici直採用。
- キャッシュを自前実装
  競合・破損耐性・並行性を考えると既存の堅牢実装を使う方が速い。cacacheはnpmでも利用され実績がある。

---

# 影響 / 帰結

良い点

- jsonlにより、POI大量出力でもパイプで破綻しにくい（メモリ効率・逐次処理）。
- エラー分類と終了コードで、AIエージェントのリトライ/分岐が定型化できる。
- Undici＋MockAgentでHTTP境界のTDDがやりやすい。
- cacacheにより反復呼び出しが速く、上流負荷も下げられる。

注意点

- json/geojson/jsonlの出力スキーマを「破壊的変更しない」運用が必要（メジャーバージョン戦略を別ADRで定義推奨）。
- Overpassレスポンスの全パターンをストリーミング変換するには、MVPでも実装難度が上がるため、段階導入（段階A→B）で進める。

---

# 実装チェックリスト（MVPで必達）

- commanderでサブコマンド実装（help/usage/例文）
- `--format`（json/geojson/jsonl）・`--within`（地名/@file/-）の共通化
- 終了コード表とJSONエラーの固定
- undiciでNominatim/Overpass/Valhallaアダプタ（timeout/retry/分類）
- poi fetch の jsonl 出力（逐次書き出し）
- cacache導入（geocode/aoi/overpass応答のキャッシュ）
- Vitestでユニット/アダプタ/CLI統合テスト一式
